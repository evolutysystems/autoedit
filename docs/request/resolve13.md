# resolve13 — ローカルLLM導入の効果分析・有償配布可否 提案書

## 0. 本書の位置づけ
`docs/request/request13.md` の問いに対する **提案書（兼・簡易設計方針書）** です。CLAUDE.md の方針（いきなり実装しない／既存実装を破壊しない／ハードコード禁止・設定は setting.json 管理／不要なライブラリを追加しない）に従い、本書はあくまで方針提示までとし、実装はレビュー後に別途 request 化します。

request13 の問いは3点です。
1. 現状処理にローカルLLMを入れるなら、**どの処理に介入させるのが最も効果があるか**。
2. そのLLM使用を前提としたアプリを、**有償で配布してよいか**。
3. **デメリット** も記載する。

> **免責（重要）**: 本書 §6 のライセンス記述は法的助言ではありません。モデル・ランタイムのライセンスは改訂されるため、実配布の直前に各提供元の最新ライセンス原文を必ず再確認し、有償配布時は法務確認を推奨します（§10）。

## 1. 現状処理の整理（LLM介入候補の洗い出し）

パイプライン実測（`src/pipeline/pipeline_runner.py` / `src/modules/*`）に基づく工程一覧です。

| # | 工程 | 実装 | 入出力の性質 | LLM適性 |
|---|---|---|---|---|
| A | 音量解析・カット閾値確認 | `volume_analyzer` + GUIダイアログ | 音声の音量(dB)解析 | ×（数値信号処理） |
| B | 無音カット | `silence_cutter`（FFmpeg silencedetect） | 音声波形→区間 | ×（波形処理、テキスト無し） |
| C-1 | 音声認識 | `WhisperTextSource`（faster-whisper） | 音声→`[{start,end,text}]` | △（既にASRが担当） |
| C-2 | **字幕テキスト後処理** | `remove_fillers`（単純置換）＋ `wrap_lines`（BudouX改行） | **テキスト→テキスト** | **◎（最有力）** |
| C-3 | 表示タイミング整形 | `adjust_display_timing` | 数値（秒）調整 | ×（規則ベースで十分） |
| C-4 | 役割判定（配信者/サブ/コメント） | `_review_timeline` で**人手**選択 | テキスト→ラベル | ○（自動化候補） |
| C-5 | 手動レビュー編集 | `subtitle_editor_dialog`（人が誤字修正・取捨選択） | 人手 | ◎（**LLMで人手削減**） |
| D | OP/ED結合 | `concat_processor` | 動画連結 | ×（メディア処理） |
| E | 出力 | `output_writer` | ファイル配置 | × |

**着眼点**: 現状 `remove_fillers` は固定リストの単純文字列置換（`subtitle_generator.py` L407-411）にすぎず、改行は BudouX の文節分割（意味理解なし）のみです。**Whisper の誤変換・句読点欠落・話し言葉の冗長さ・固有名詞ゆれ**はすべて手動レビュー画面（C-5）でユーザーが直しており、ここが唯一の「人手が残るボトルネック」です。

## 2. 結論（先出し）

- **最も効果が大きいのは C-2：字幕テキストの後処理（ASR誤り修正・整形）にローカルLLMを介入させること**です。理由は §3。
  - 副次的に **C-4 役割の自動推定**、および**新機能としてチャプター/概要文の自動生成**が候補（§3.3）。
  - **B 無音カットや A 音量解析への介入は非推奨**（テキストでなく信号処理であり、LLMの土俵ではない）。
- **有償配布は「条件付きで可能」**です。鍵は**採用モデルのライセンス**（§6）。Apache-2.0 / MIT の日本語対応モデルを選べば商用配布のハードルは低く、Llama系・Gemma系は表示義務・利用規約(AUP)・同梱条件などの遵守が必要です。
- ただし**デメリットは小さくない**（処理時間増・配布物肥大化・ハルシネーション事故・環境依存クラッシュ・決定論性喪失）。§7。
- **導入するなら「既定OFFのオプション機能」**として、既存のレビュー画面（C-5）を残したまま「下書きをLLMが用意し、人が最終確認」する形が最も安全（§5・§8）。

## 3. どの処理にLLMを介入させると最も効果があるか

### 3.1 推奨（第一候補）: 字幕テキストの後処理（C-2）
Whisper 出力を、焼き込み前に LLM で一括整形します。タイムスタンプ（`start`/`end`）は保持し、**`text` だけを書き換える**ため、既存アーキテクチャ（`timeline = [{start,end,text}]` を各工程が受け渡す構造）にそのまま乗ります。

LLM が担える具体タスク:
- **誤変換・同音異義語の文脈補正**（例:「以外」↔「意外」、固有名詞のゆれ統一）
- **フィラー・言い淀みの自然な除去**（固定リスト置換より文脈依存で高精度）
- **話し言葉→読みやすいテロップへの軽整形**（過度な冗長表現の圧縮。※意味は変えない前提）
- **句読点・意味に沿った改行位置の提案**（BudouX の文節改行を補完/代替）

効果が大きい理由:
1. **現状ここだけ人手が残る**（C-5 手動レビュー）。LLM下書きで編集量を削減でき、体感価値が最大。
2. **入出力が純テキスト**でLLMの得意領域。既存の `TextSource` 抽象の**後段**に差し込むだけで責務分離が保てる。
3. 既存の**フォールバック思想と相性が良い**（LLM失敗・タイムアウト時は元テキストで続行すればテロップ全損しない。VAD失敗時フォールバック等と同じ設計＝`subtitle_generator.py` L398-405）。

### 3.2 次点: 役割の自動推定（C-4）
現在は配信者/サブ/コメントを人が手動選択（`subtitle_editor_dialog.py`）。LLMに文体・一人称・文脈を渡し**役割の初期値を推定**させ、人は確認・修正のみに軽減できます。ただし精度が読めず、誤ラベルは色分けの事故になるため**「初期値の提案」に限定**すべき。

### 3.3 付加価値（新機能寄り）: チャプター/概要/タイトル生成
確定字幕（全文＋時刻）から、**チャプター区切り・動画概要文・タイトル候補・ハッシュタグ**を生成。編集後の「投稿準備」を助ける機能で、有償版の差別化ポイントになり得ます。パイプライン本流を汚さず、出力後の任意ステップとして追加可能。

### 3.4 非推奨: 無音カット・音量解析（A/B）
これらは音声波形/音量(dB)の**信号処理**であり、テキストLLMの適用対象ではありません。介入しても効果がなく、処理を複雑化・低速化させるだけです。

## 4. 推奨アーキテクチャ案（既存構造への載せ方）

> 実装は本書レビュー後。ここでは方針のみ。既存を破壊せず、**抽象1つ＋設定セクション1つ**の追加に閉じます。

### 4.1 差し込み位置
`subtitle_generator.run()`（`subtitle_generator.py` L732〜）の **`text_source.extract()` の直後、`adjust_display_timing` / `_review_timeline` の前**に後処理フックを1段追加します。

```
extract(ASR) → [新] TextPostProcessor.process(timeline) → adjust_display_timing → _review_timeline(人が最終確認) → burn
```

### 4.2 抽象クラス案
既存 `TextSource` と同じ流儀で `TextPostProcessor` 抽象を新設し、既定実装を選択可能にします。

```
TextPostProcessor（抽象）
  process(timeline, subtitle_cfg) -> timeline   # start/end は不変、text のみ整形
  ├─ NullPostProcessor        … 既定。何もしない（＝現行挙動、後方互換）
  └─ LocalLlmPostProcessor    … ローカルLLMで整形（オプション）
```

`resolve_text_postprocessor(settings)` を `resolve_text_source` と同様に用意し、無効/未導入時は `NullPostProcessor` を返して**現行挙動を完全維持**します。

### 4.3 実行方式の比較（追加ライブラリ最小化の観点）
CLAUDE.md「不要なライブラリを追加しない」を踏まえ、**外部プロセス/HTTP方式を第一候補**とします。

| 方式 | 追加依存 | 長所 | 短所 |
|---|---|---|---|
| **A. Ollama等ローカルサーバをHTTP呼出**（推奨） | 追加pkgほぼ不要（標準 `urllib` でも可） | 本体を汚さない・モデル管理を委譲・導入容易 | ユーザーがランタイムを別途用意/同梱が必要 |
| B. `llama-cpp-python` を**任意依存**として遅延import | 任意（未導入時はNullへフォールバック） | 単一プロセスで完結 | ビルド/環境差が大きい・配布が重い |
| C. モデル同梱＋自前ローダ | 大 | オフライン完結 | 配布物肥大・保守コスト大 |

いずれも **faster-whisper と同じ「任意依存・未導入時は安全にスキップ」パターン**（`subtitle_generator.py` L323-330）を踏襲します。

### 4.4 setting.json 拡張案（追加のみ・既定は現行維持）
`subtitle` セクションに、または新規 `subtitle_llm` セクションとして追加。**`enabled` 既定 `false`** で後方互換を担保します。

```jsonc
"subtitle_llm": {
  "enabled": false,              // 既定OFF＝現行挙動を一切変えない
  "backend": "ollama",           // "ollama" | "llama_cpp" | "none"
  "endpoint": "http://127.0.0.1:11434",
  "model": "",                   // 例: Apache-2.0/MIT系の日本語対応モデル名
  "temperature": 0,              // 決定論性のため 0 を既定
  "timeout_sec": 30,             // 超過時は元テキストで続行（フォールバック）
  "max_chars_per_call": 2000,    // 長尺は分割
  "tasks": {                     // 介入内容を個別ON/OFF
    "fix_typos": true,           // 誤変換補正
    "remove_fillers": true,      // フィラー除去（既存の単純置換を置換/補完）
    "punctuation": true,         // 句読点補完
    "rewrite": false             // 話し言葉の圧縮（意味改変リスクのため既定OFF）
  }
}
```

### 4.5 安全設計（既存の設計思想に合わせる）
- **決定論性**: `temperature=0`。同一入力→同一出力を保ち、テスト・サポートを容易にする。
- **ハルシネーション対策**: プロンプトで「**内容・語順を変えず、明らかな誤変換と表記のみ修正。新規情報の追加・要約は禁止**」を明示。C-5 の手動レビュー画面は**必ず残す**（人の最終確認を担保）。
- **フォールバック**: LLM未導入/接続失敗/タイムアウト/空応答時は**元の timeline をそのまま返す**（テロップ全損を回避。既存の VAD/ BudouX フォールバックと同思想）。
- **ログ**: 既存 `logger` に整形前後の差分件数を INFO 出力。

## 5. 有償配布は可能か（ライセンス観点）

**結論: 条件付きで可能。** 最大の分岐は「どのモデルを使い、どう配るか」です。

### 5.1 モデル重み（weights）のライセンス — 最重要
| モデル系統 | 代表ライセンス | 商用配布 | 主な条件 |
|---|---|---|---|
| Qwen2.5 等 | Apache-2.0（版により差） | ○ 容易 | 表示・NOTICE程度 |
| Mistral（Apache版） | Apache-2.0 | ○ 容易 | 同上 |
| Phi 系 | MIT | ○ 容易 | 同上 |
| Llama 3.x 系（ELYZA等の派生含む） | Meta Llama Community License | △ 可だが条件付き | 「Built with Llama」表示、AUP遵守、月間7億ユーザー超は別途、派生命名規則、ライセンス同梱 |
| Gemma 系 | Google Gemma Terms | △ 可だが条件付き | 使用禁止ポリシー遵守、条件の下流伝播、通知同梱 |

> **推奨**: 有償配布のライセンスリスクを最小化するなら、**Apache-2.0 / MIT の日本語対応モデル**を選定。Llama/Gemma 系を採用する場合は各条件（表示義務・AUP・同梱）の遵守を配布フローに組み込む。**版ごとにライセンスが異なる**ため、採用モデルの当該バージョン原文を必ず確認。

### 5.2 推論ランタイムのライセンス
- **Ollama**: MIT（配布容易）。**llama.cpp**: MIT。→ 商用配布に支障なし。
- 既存: **faster-whisper**(MIT) / **CTranslate2**(MIT) / **Whisperモデル**(MIT) / **BudouX**(Apache-2.0)。→ 現状の音声認識まわりは商用フレンドリー。
- **PySide6**: LGPL。動的リンク・改変非配布の通常利用なら商用可（既に採用中の論点）。

### 5.3 既存の別論点: FFmpeg 同梱
本リポジトリは `src/ffmpeg/ffmpeg.exe` を**同梱**しています。FFmpeg は**ビルド構成により GPL/LGPL が変わり、GPL 版同梱は配布物全体へのGPL波及議論**が生じます。これは**LLMとは独立した既存の確認事項**ですが、**有償配布の法務確認時に併せて点検**すべきです（同梱ビルドのライセンス・ソース提供義務の有無）。

### 5.4 生成物（出力字幕・動画）の権利
- 多くのモデルは**出力の商用利用を制限しない**が、一部（Llama等）は「出力で競合LLMを訓練しない」等の条項あり。エンドユーザー動画の権利は通常ユーザーに帰属。→ 利用規約に明記推奨。

### 5.5 推奨する配布形態
1. **モデルは本体に同梱せず、初回起動時にユーザー同意の上で取得**（＝配布物肥大とライセンス波及の双方を軽減）。または「ユーザーが用意した Ollama を指すだけ」構成。
2. **各種ライセンス文書・NOTICE を同梱**し、必要な帰属表示（例:「Built with 〇〇」）を about 画面等に表示。
3. **LLM整形は既定OFFのオプション**とし、無効時はモデル不要で従来通り動作（無料/軽量エディションと有料エディションの切り分けにも使える）。
4. 有償配布・EULA・特商法/返金等は**法務確認**の上で整備。

## 6. デメリット（総覧）

| 区分 | 内容 | 緩和策 |
|---|---|---|
| **処理時間** | 本アプリは CUDA 不使用方針（resolve8＝CPU実行固定）。CPUでのLLM整形は字幕全体で数分〜数十分になり得る | 既定OFF、対象を短文の整形に限定、分割・タイムアウト、必要な人のみ有効化 |
| **配布物肥大** | モデルは数GB規模。同梱すると配布・更新が重い | モデル非同梱＋初回DL方式（§5.5-1） |
| **品質/事故** | LLMが内容を改変・創作（ハルシネーション）。字幕は「発話の忠実な記録」であり意味改変は事故 | 「修正のみ・追加禁止」プロンプト、`temperature=0`、C-5 手動レビュー必須、`rewrite` 既定OFF |
| **環境依存** | RAM 8〜16GB+要求。低スペック環境でOOM/クラッシュ | 事前チェック、失敗時フォールバック、推奨環境の明示 |
| **決定論性喪失** | 出力が揺れるとテスト・再現・サポートが困難 | `temperature=0`、入出力ログ保存 |
| **依存・保守** | ランタイム/モデル追加は「不要ライブラリを増やさない」方針と緊張。バージョン更新追随コスト | 任意依存＋外部プロセス方式（§4.3-A）、未導入時Nullフォールバック |
| **ライセンス/法務** | 有償配布はモデル/ランタイム/FFmpegの条件遵守が必須。違反は事業リスク | Apache/MIT系選定、文書同梱、法務確認（§5） |
| **サポート範囲拡大** | ユーザー環境の多様化で問い合わせ増 | 推奨構成の固定、診断ログ |

## 9. まとめ
- **どこに入れるか**: 迷わず **C-2 字幕テキスト後処理**（既存の唯一の人手ボトルネックを削減、テキスト入出力でLLM適性最大、既存抽象に載せやすくフォールバックとも整合）。役割推定・概要生成は次点。無音カット/音量解析は非推奨。
- **有償配布**: **条件付きで可能**。Apache-2.0/MIT系モデルを選び、モデル非同梱DL＋文書同梱＋法務確認を行うのが安全。
- **デメリット**: 処理時間・配布肥大・ハルシネーション事故・環境依存・決定論性・法務コスト。ゆえに**既定OFFのオプション＋人手レビュー維持**を強く推奨。
