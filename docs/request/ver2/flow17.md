# flow17 — Twitch アーカイブ切り抜き 段階リリースフロー

## 0. 本書の位置づけ
`docs/request/ver2/resolve17.md`（要望設計書・回答反映済み）を、**安全に少しずつ本番配布していくためのリリース計画**に落とした手順書です。
- 実装の設計根拠は resolve17.md を参照（本書は「どの順で・何を含めて・どうリリースするか」に集中）。
- 実際のビルド／配布の正典手順は `docs/HowToRelease.md`（＋`installer/README.md`）。本書はそれを各段階へ適用する。
- CLAUDE.md 方針（既存を壊さない／CPU-only 維持／不要な依存を足さない／設定は setting.json）を全リリースで遵守。

現行版は **v1.0.9**。本サブシステムは複数の **マイナーリリース（v1.1.0 → v1.5.0…）** に分割して配布する。

## 1. リリース戦略の原則

本アプリは `updater.py` により **GitHub Releases を検知して全ユーザーへ自動更新通知**が飛ぶ（`main_window.start_update_check`）。つまり **中間リリースも全ユーザーに届く**。よって以下を全段階で厳守する。

1. **回帰ゼロが最優先**：どのリリースも **現行「クリップ用」機能（現行 `MainWindow` フロー）を一切劣化させない**。各リリースの受入基準トップは「現行動画で従来と同一出力」（`HowToRelease.md §6` チェックリスト）。
2. **未完成機能は露出させない（機能フラグ）**：アーカイブタブは **その段階で完成した機能だけを表示**し、未完成部は「準備中」表示 or 非表示にする。制御は `archive.enabled` と段階別の内部フラグで行い、**半完成のUIをユーザーに触らせない**。
3. **縦スライスで“毎回動く”**：各リリースは「入力 → 何らかの結果 → 出力」まで**通しで完結**する最小単位にする（画面だけ・裏だけ、は出さない）。
4. **local 先行 → Twitch 後付け**：ネットワーク/認証の複雑さを切り離すため、まず `downloader:"local"`（手元 mp4 指定）で採点系を固め、Twitch 自動取得は後段リリースで載せる（resolve17 §4.3.1 / §9）。
5. **依存は確定分だけ**：追加してよいのは **twitch-dl（同梱）** と **PySide6.QtCharts** のみ。**torch/ML/OpenCV/MediaPipe/YOLO/CUDA は入れない**（resolve17 §8 確定）。
6. **バージョンは version.py と .iss を必ず同時更新**（`src/version.py` の `__version__` と `installer/AutoEdit.iss` の `MyAppVersion` を一致。`version.py:2-3` 明記）。

## 3. 各リリースの詳細

各節の見出し語：**含む／含まない／依存・spec／setting.json／露出制御／受入基準／後方互換**。

### R0 — v1.1.0：メイン画面タブ化の土台
- **含む**：`MainWindow` を `QTabWidget` 化。現行 UI を `ClipTabWidget` へ**無改変移設**（resolve17 §4.2）。「アーカイブ切り抜き用」タブは**プレースホルダ（「準備中」表示）**。
- **含まない**：採点・取得・結果画面・切り抜き（一切のアーカイブ機能）。
- **依存・spec**：追加なし。`main_window.spec` 変更なし。
- **setting.json**：`archive.enabled` のみ先行追加可（既定 false でUIは準備中固定）。他キーは未追加でよい。
- **露出制御**：アーカイブタブは中身「準備中」。誤操作でパイプラインが走らないこと。
- **受入基準**：
  - [ ] 現行「クリップ用」タブが**従来と完全同一**に動作（参照→設定→実行→字幕編集→出力、D&D、自動更新）。
  - [ ] 現行動画で**焼き込み結果が従来と一致**（回帰ゼロ）。
  - [ ] `HowToRelease.md §6` の全項目 PASS。
- **後方互換**：完全互換（UIの器が変わるのみ）。**リスク最小の先行リリース**として土台を安全に本番投入。

### R1 — v1.2.0：方式A中核採点（local）＋切り抜き焼き込み
- **含む**：
  - アーカイブタブに **local mp4 入力**（`downloader:"local"`）＋「採点開始」。
  - **方式A・中核スコアラ**（`comment`〔※local時は取込 or 0〕/`volume`/`silence`/`speech_rate`、resolve17 §4.4.1）＝**torch 不要のヒューリスティック**。
  - 窓生成（5分/1分スライド）→ イベント統合 → **TOP5**（§4.5）。
  - **簡易結果 UI**（TOP5 区間と点数の一覧＋使用可否）＋「完了」。
  - **切り抜き＋字幕焼き込み**（既存 `build_subtitle_file`/`burn_subtitle` 流用、縦/横対応、出力名先頭 `archive`、§4.8）。
- **含まない**：Twitch 取得・ログイン、QtCharts グラフ、プレビュー再生、任意スコアラ、方式B。
- **依存・spec**：追加なし（**新規依存ゼロ**）。`main_window.spec` 変更なし。
- **setting.json**：`archive`（`download.downloader:"local"`, `scoring`(方式A), `output.clip_prefix:"archive"`）を追加（resolve17 §4.9）。※`auth`/`llm` はキーだけ既定値で入れてよい（未使用）。
- **露出制御**：方式は A 固定表示（B は非表示 or 選択不可）。コメントスコアは「chat JSON 取込時のみ有効」と明示（未取込は音声採点のみ）。
- **受入基準**：
  - [ ] R0 の回帰ゼロ項目（毎回）。
  - [ ] local mp4 で採点が完走し、TOP5 が妥当に出る（実素材）。
  - [ ] 完了で TOP5 クリップが生成され、字幕が焼き込まれる（縦/横・色/役割/個別フォントが現行仕様どおり）。
  - [ ] 出力ファイル名が `archive` 接頭辞。
- **後方互換**：現行機能は不変。`archive` は追加のみ（`_merge_with_defaults` で旧設定も自動補完、§7）。

### R2 — v1.3.0：結果画面フル（グラフ／字幕修正／プレビュー）
- **含む**：resolve17 §4.7 の `ArchiveResultWindow`。
  - 上：**QtCharts 採点グラフ**（縦=点数/横=時間、TOP5 マーカ、§4.7.1）。
  - 下左：**字幕修正**＝既存 `SubtitleEditorDialog` を `SubtitleEditorWidget` に切り出して埋込（§4.7.2、**ダイアログ版の既存呼び出しは不変**）。
  - 下右：**プレビュー**（QtMultimedia、選択区間シーク。不可環境は ffmpeg 静止画フォールバック、§4.7.3）。
  - R1 の簡易 UI を本画面へ差し替え。
- **含まない**：Twitch 取得・ログイン、任意スコアラ、方式B。
- **依存・spec**：**`PySide6.QtCharts` 追加**。`main_window.spec` に QtCharts（と QtMultimedia）を同梱（`HowToRelease.md §4` の spec へ追記）。**CUDA/torch は従来どおり非同梱**。
- **setting.json**：変更なし（グラフは QtCharts 固定）。
- **露出制御**：プレビュー不可環境では静止画に自動フォールバック（§5）。
- **受入基準**：
  - [ ] R0 回帰ゼロ／R1 の採点・焼き込みが本画面経由でも成立。
  - [ ] グラフに窓スコアと TOP5 マーカが表示され、クリックで下段が同期。
  - [ ] 字幕修正の編集結果が焼き込みに反映（既存 `result_items()` 互換）。
  - [ ] **クリーンPC**でプレビュー（QtMultimedia）が再生 or 静止画フォールバックする（§8-11 検証）。
- **後方互換**：`SubtitleEditorDialog` の I/F 非破壊（§7）。QtCharts はビルド同梱のみでユーザー操作不変。

### R3 — v1.4.0：Twitch ログイン＋自動取得（VOD＋コメント）
- **含む**：resolve17 §4.3 / §4.3.0。
  - アーカイブタブに **「Twitch ログイン」**（OAuth 認可コードフロー、`twitch_auth.py`）。
  - **自分の VOD のみ許可**：Helix で所有判定、不成立 URL は実行不可（§8-1 確定）。ログイン時は自分の VOD 一覧提示。
  - **twitch-dl 同梱**で **VOD=`download` / コメント=`chat json`** を取得 →正規化（§4.3.1/4.3.2）。
  - コメントスコア（ｗ数・急増）が実データで有効化 → **方式A が完全体**。
- **含まない**：任意スコアラ、方式B。
- **依存・spec**：**twitch-dl CLI を同梱**（`HowToRelease.md §4` spec の datas/バイナリへ）。Client-ID は設定化。**torch/YOLO/CUDA なし**。
- **setting.json**：`archive.download.downloader:"twitch-dl"`、`twitch_dl_path`、`chat_source:"twitch-dl"`、`auth`(client_id/redirect_port/owner_only) を有効化（§4.9）。`local` も残す。
- **露出制御**：未ログイン時は取得不可の案内。他者 VOD は明示エラーで拒否。
- **受入基準**：
  - [ ] R0 回帰ゼロ／R1・R2 の採点・結果画面・焼き込み。
  - [ ] ログイン成立 → **自分の VOD**で取得→採点→切り抜きが通し完走。
  - [ ] **他者 VOD の URL は実行拒否**される。
  - [ ] コメント取得失敗時もコメントスコア0で採点継続（§5）。
  - [ ] （検証）自分の sub-only VOD でのトークン両立（§8-3b：不可ならトークン貼付導線）。
- **後方互換**：取得は追加経路。`local` 指定で従来どおり動作。ログイン情報の保存はローカルに限定（機微情報の扱いに注意）。

### R4+ — v1.5.0〜：任意スコアラ／方式B の段階追加
- **方針**：resolve17 §4.4.2 の各スコアラを **1リリース1機能**で追加。すべて **既定 OFF**（`archive.scorers.*:false`）で、依存が無ければ自動スキップ（§3-3）。
- **順序の目安**（軽い順・依存とリスクで判断）：
  1. **固有名詞**（ヒューリスティック、依存なし）／コメント特徴の拡充（草率・!?率・初見率）。
  2. **感情/笑い/ピッチ**（ML 系）→ **導入可否は要再判断**（§8Q-4：torch 同梱は +0.5〜1GB。CPU-only 方針と衝突。まず軽量近似で検討）。
  3. **映像（OpenCV/顔）**→ 実行時間大。方式B の一部として。
  4. **外部 LLM**（§4.6.2、HTTP。同梱しない）。
  5. **方式B**（30秒特徴→メモリ時系列→5分積分→ランキング→LLM）を統合（§4.6）。
- **各リリース共通の含まない**：**YOLO ゲームイベントは当面不採用**（§8Q-5：実行時間・依存が過大）。
- **依存・spec**：機能ごとに **その回だけ**依存を足すか判断。足す場合のみ spec 更新（重い ML は原則見送り）。
- **受入基準**：各回とも「**その機能 OFF で従来と完全同一**」＋「ON で期待どおり寄与」を確認。**既定 OFF なので一般ユーザーへの影響ゼロ**を担保。

## 4. 各リリース共通の作業手順（HowToRelease 連携）

すべてのリリース（R0〜）で、実装・レビュー完了後に以下を実施する。手順の正典は `docs/HowToRelease.md §10` と `installer/README.md`。

1. **バージョン更新（必須の2点同時）**
   - `src/version.py` の `__version__` を更新（例 `1.1.0`）。
   - `installer/AutoEdit.iss` の `MyAppVersion` を**同値**に（`version.py:2` の指示どおり不一致厳禁）。
2. **リリースノート作成**：`installer/release_notes_v<ver>.md`（既存 `release_notes_v1.0.9.md` に倣う）。新機能/修正・**動作要件**・既知の不具合・SHA-256・**「アーカイブ機能は段階公開中／未実装部は準備中」**の注記。
3. **クリーン仮想環境でビルド**（`HowToRelease.md §2/§5.1`）：`nvidia-*-cu12` は入れない（CPU-only）。R2 以降は `PySide6.QtCharts`、R3 以降は `twitch-dl` 同梱を spec に反映。
4. **FFmpeg 同梱＆配布レイアウト**（`§5.2/5.3`）。
5. **本体 zip 化＋SHA-256**（`§7.3`）。2GiB 超なら分割（`installer/README.md §1.1`）。QtCharts/twitch-dl 追加後のサイズを確認。
6. **インストーラ生成**（`.iss` の #define：バージョン／`PayloadParts`／`PayloadSHA256` → ISCC、`installer/README.md §2-3`）。
7. **GitHub Releases 公開**（`gh release create/upload`、`installer/README.md §4`）→ これにより `updater.py` が**全ユーザーへ自動更新通知**。
8. **homepage ダウンロード URL 差し替え**（`Dev.tsx`、別リポジトリ、`installer/README.md §5`）。
9. **クリーンPC で動作確認**（`HowToRelease.md §6` ＋ 本書 §3 の当該リリース受入基準）。**回帰ゼロを必ず確認**。
10. （推奨）**コード署名**（`§7.2`）。
11. Git：機能ブランチ → レビュー → `main` へマージ → タグ `v<ver>`（コミットは main 直コミットせずブランチ運用）。

> **公開前ゲート**：自動更新で全員に届くため、**§3 の受入基準と回帰ゼロが未達なら公開しない**。未完成機能が露出していないか（機能フラグ）を公開前に必ず確認する。

## 5. バージョニング方針
- セマンティックバージョニング（`vMAJOR.MINOR.PATCH`、`HowToRelease.md §8`）。
- 本サブシステムの各段階は **MINOR 増**（v1.1.0 → v1.2.0 …）。段階内の不具合修正は **PATCH 増**（例 v1.2.1）。
- `version.py` / `AutoEdit.iss` / zip 名 / 配布ページを一致させる。

## 6. ロールバック・リスク対応
- **自動更新で不具合が拡散するリスク**：各段階は機能フラグで隔離。問題時は **`archive.enabled=false` 相当の無害化パッチ（PATCH リリース）** で素早く塞ぐ／直前バージョンのインストーラを再掲。
- **依存追加の肥大化**（R2 QtCharts / R3 twitch-dl）：公開前に配布サイズと GitHub Releases 上限（2GiB/ファイル）を確認（`HowToRelease.md §7.3`）。
- **CPU-only 維持**：R4+ の ML 系は容量・実行時間の実測後に導入可否を再判断（§8Q-4/5）。安易に torch を足さない。
- **機微情報**（Twitch トークン）：ローカル保存に限定。ログへ出力しない。
