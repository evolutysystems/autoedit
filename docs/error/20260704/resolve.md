# Resolve — 「未確認 806202.crdownload」でダウンロードが完了しない問題

## 1. 結論（要約）

友達の環境で発生している `未確認 806202.crdownload`（英語UIでは `Unconfirmed 806202.crdownload`）は、
**ブラウザ（Chrome / Edge）がダウンロードを途中で止めている状態** を表します。

- `.crdownload` … Chrome/Edge がダウンロード中に使う **一時ファイル** の拡張子。
  完了すると本来の名前（`AutoEditSetup.exe`）にリネームされる。
- `未確認 NNNNNN`（Unconfirmed NNNNNN）… まだ最終ファイル名が確定していない状態の仮の名前。
- つまり **ダウンロードが「完了せずに止まった / ブロックされた」まま** になっている。

自分の環境で成功し、友達の環境だけで失敗している最大の要因は、
**配布している `AutoEditSetup.exe` がコード署名（デジタル署名）されていない実行ファイルであるため、
ブラウザの安全機能（Google セーフ ブラウジング / Microsoft SmartScreen）がダウンロードを警告・ブロックしている** ことです。
（開発者本人の環境はダウンロード履歴・実行履歴があり警告が出にくい／既に例外扱いになっているため再現しません。）

---

## 2. 原因の詳細

### 原因A（最有力）: 署名なしEXEに対するブラウザのブロック

`installer/AutoEdit.iss` を見ると、生成物は以下です。

- `OutputBaseFilename=AutoEditSetup` → `AutoEditSetup.exe`
- コード署名の設定なし（`SignTool` 指定なし）、`SetupIconFile` も未設定（69行目コメント）

署名のない `.exe` を GitHub からダウンロードすると、
Chrome / Edge は「この種類のファイルは PC に損害を与える可能性があります」と判断し、
ダウンロードを **保留（未確認のまま）** にします。ユーザーが「保持する」を押さない限り
`.crdownload` のまま完了しません。

特に Chrome の **セーフ ブラウジング（保護強化機能 / Enhanced Protection）** が有効な環境では、
未知の署名なしEXEは自動でブロックされやすく、友達の環境がこれに該当している可能性が高いです。

### 原因B: ダウンロードの中断（ネットワーク）

`806202` はダウンロード済みバイト数（約 806KB）を示す仮名の可能性があります。
ネットワーク切断・回線不安定・ウイルス対策ソフトの割り込みでダウンロードが途中停止した場合も
`.crdownload` が残ります。GitHub Releases は数百MB〜GB級の配布のため、
不安定回線では途中で失敗しやすい点も要因になり得ます。

### 原因C: ウイルス対策ソフト / 企業ポリシーによる遮断

サードパーティのセキュリティソフトや、会社・学校の端末ポリシーが
署名なしインストーラーの取得を遮断しているケースもあります。

---

## 3. 対応方法

### 3-1. 友達がすぐ実行できる回避策（暫定）

まずはこれで完了できるか確認してください。

1. **ダウンロードを「保持」する**
   - Chrome: 画面右上のダウンロード一覧を開き、ブロックされた項目の「︙」→ **「保持する」**（Keep）。
   - Edge: ダウンロード通知の「…」→ **「保持」** → 「詳細表示」→ **「保持する」**。
2. **完了後、SmartScreen 警告を通過する**
   - 実行時に「WindowsによってPCが保護されました」→ **「詳細情報」→「実行」** を押す。
3. うまくいかない場合は **別ブラウザ（Firefox 等）や別回線** で再ダウンロードを試す。
4. ダウンロードが途中で止まる場合は、既存の `.crdownload` を削除してから **再ダウンロード**。

> 注意: 上記はあくまで暫定回避です。一般の利用者に毎回「保持」「詳細情報→実行」を操作させるのは
> 配布品質として不十分なため、下記の恒久対策を推奨します。

### 3-2. 恒久対策（配布側で行う）

**優先度: 高 — コード署名（Authenticode）の導入**

`AutoEditSetup.exe` に **コード署名証明書** で署名するのが根本解決です。

- OV（Organization Validation）証明書 … 安価だが SmartScreen の信頼（レピュテーション）蓄積に時間がかかる。
- EV（Extended Validation）証明書 … 発行直後から SmartScreen の警告が出にくい（推奨）。
- Inno Setup では `SignTool` ディレクティブ、または `signtool.exe` で
  ビルド後の `AutoEditSetup.exe` を署名する。

例（署名コマンド）:
```
signtool sign /fd SHA256 /tr http://timestamp.digicert.com /td SHA256 ^
  /f cert.pfx /p <password> installer\Output\AutoEditSetup.exe
```

**優先度: 中 — ZIPでラップして配布**

署名導入までの間、`AutoEditSetup.exe` を **ZIPに圧縮して配布** すると、
ブラウザの「危険な実行ファイル」判定を回避しやすくなります。
利用者はZIP展開後にインストーラーを実行します（実行時のSmartScreen警告は残る）。

**優先度: 中 — SHA256 と手順の明示**

配布ページ（GitHub Releases の説明 / README）に以下を明記し、利用者が安心して「保持」できるようにします。

- `AutoEditSetup.exe` の SHA256 ハッシュ（`Get-FileHash` で取得）
- 「ブラウザに警告が出た場合の保持手順」「SmartScreen で 詳細情報→実行 する手順」
- 署名導入までは警告が出る旨の但し書き

**優先度: 低 — 配布経路の見直し**

長期的には Microsoft Store / winget など、署名・審査済みの配布チャネルの利用も検討可能。

## 4. まず切り分けるべきこと（原因確定のための確認）

友達に以下を確認すると、原因A / B / C のどれかが特定できます。

1. ダウンロード一覧に **「危険なファイル」「ブロックされました」等の警告表示** があるか？
   → あれば **原因A（署名なしブロック）** で確定。3-1 の「保持する」で通過可能。
2. 警告はなく、途中でバイト数が止まる／エラーになるか？
   → **原因B（ネットワーク中断）**。別回線・再ダウンロードで確認。
3. セキュリティソフト（ウイルスバスター等）の通知が出ていないか？
   → **原因C**。当該ソフトの例外設定で確認。

---

## 5. 推奨アクション（まとめ）

| 対象 | 対応 | 位置づけ |
|------|------|----------|
| 友達（今すぐ） | ダウンロードを「保持する」→ 実行時「詳細情報→実行」 | 暫定回避 |
| 配布側（根本） | `AutoEditSetup.exe` を **コード署名（EV推奨）** | 恒久対策・最優先 |
| 配布側（つなぎ） | EXEを **ZIP化して配布** ＋ **SHA256と手順を明記** | 署名導入までの緩和策 |

> 本問題は AutoEdit アプリ本体のバグではなく、
> **「署名のない配布EXEに対するブラウザ／OSの安全機能」による挙動** です。
> コード署名の導入で解消できます。
