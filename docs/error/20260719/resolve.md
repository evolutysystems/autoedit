## 1. 事象

3時間程度のアーカイブ動画を1クリック実行したところ、出力に以下が発生した。

- 動画の最初から、**映像**と**（音声・テロップ）**がズレている。
- **映像が先に流れ**、**音声・テロップが後に流れる**（映像が先行）。
- **音声とテロップは互いにズレていない**（両者は一致）。
- 発生したのは開発機（本PC）ではなく、**別PC（NVIDIA GPU 内蔵）で使用**した場合。

この「**映像 vs (音声＋テロップ) がズレるが、音声とテロップは一致する**」という切り分けが、原因特定の決定的な手掛かりである（§2-2）。

---

## 2. 原因調査

### 2-1. 該当設定と通過経路

`setting.json` は本事象条件下で以下：

- `silence_cut.enabled=true` / `extract_mode="seek"` / `concat_reencode=false`
  → 無音カットは **`cut_and_concat_seek`**（seek抽出→`_concat_demux` の `-c copy` 連結）を通る。
- `subtitle.enabled=true`（テロップON）→ **`burn_subtitle`** で全長を再エンコード。
- `general.opening_enabled=false` / `ending_enabled=false` → OP/ED結合はスキップ。

したがって、本事象の最終的な映像・音声の同期関係は **`cut_and_concat_seek` が作る映像PTS**と、それを **`burn_subtitle` がどう再構築するか**の2点で確定する。

### 2-2. 決定的手掛かり：「テロップは音声と一致・映像だけがズレる」の意味

テロップ（ASS）は **映像フレームへ焼き込まれる**。通常、焼き込んだテロップは映像と一体で動くはずである。にもかかわらず本事象では **テロップが（映像ではなく）音声と一致**している。これは次を意味する。

- ASS の表示時刻は、**焼き込み時点の映像PTS（時刻ラベル）** に突き合わせて描画される。
- Whisper のタイムスタンプは **音声サンプルの連続時刻**（＝実コンテンツ時刻）で付く。
- 「テロップ＝音声一致」ということは、**焼き込み後の映像PTSの目盛りは音声と揃っている**。
- それでも **映像の“絵”がズレて見える** ＝ **各フレームの PTS（時刻ラベル）と、そのフレームが本来映すべきコンテンツ（絵）が食い違っている**。

つまり **「フレームの並べ直し（PTS の付け替え）で、絵と時刻の対応が崩れた」** ことが事象の本質である。この観測は §2-4 の主因と完全に整合する。

### 2-3. 【検証で否定】ソースは VFR ではない

初版は主因を「VFRソースを CFR 化する際の dup/drop でフレーム数と実尺が乖離し、`setpts` 均等再配置が破綻する」と推定していたが、**実素材 `ark.mp4` の実測でこの前提は否定された**（`verify.md` §1）。

```
ark.mp4 : hevc(Main) 2560x1440 hev1 / 2時間0分15秒
  r_frame_rate   = 60/1
  avg_frame_rate = 59.99989
  nb_frames = 432898  vs  実尺(7214.9677s) × 60 = 432898.06   → 乖離 0.06 フレーム
  10秒窓のフレーム数 @0s/1800s/3600s/5400s/7100s = すべて 600
```

- **フレーム数＝実尺×60 が誤差 0.06 フレームで成立**しており、dup/drop 収支はゼロ。局所的な VFR も無い。
- したがって **「VFR × `setpts`」は本事象の説明にならない**。
- 併せて、初版 §2-4 の「NVIDIA GPU 搭載機＝OBS+NVENC の VFR 素材」という相関も**本件では成立しない**（NVENC 収録だが記録は CFR）。

### 2-4. 【主因】区間音声のAACフレーム量子化 × concat のファイル尺オフセットが、映像PTSに境界空白を作る

`ark.mp4` のスライスに `silence_cutter` を実コードのまま適用して実測した結果、以下の連鎖が確定した（`verify.md` §2）。

#### (1) `extract_segment` の出力は、区間ごとに**音声だけが約21.3ms長い**

| 項目 | 実測（24区間すべてで同傾向） |
|---|---|
| 映像尺 | 要求区間長 ≒ `nb_frames/60`（誤差 +0〜17ms＝フレーム量子化のみ） |
| 音声尺 | **要求区間長 +21〜22ms が全区間で一定** |
| 音声 − 映像 | **常に +約21.3ms** |

**21.3ms = 1024サンプル ÷ 48000Hz ＝ AAC 1フレーム**。AAC は 1024 サンプル単位でしかフレームを作れないため、区間の端数が必ず1フレーム分に切り上げられる。これは `-t` の指定精度や `-ss` の挙動とは無関係に、**AAC エンコーダの構造上必ず発生する**。

#### (2) `_concat_demux`（`-c copy`）は**ファイル尺＝長い方（音声）**で次区間のオフセットを進める

concat デマルチプレクサは各入力ファイルの尺ぶんだけ後続のタイムスタンプをずらす。区間ファイルの尺は「映像尺」ではなく **max(映像, 音声) ＝ 音声尺** で決まるため、**区間境界ごとに映像側だけ約21msの空白**が生じる。音声は連続したまま。

#### (3) 結果：`silence_cut.mp4` の映像PTSは「階段状」に膨張する

```
silence_cut.mp4（26.2秒ぶんの有音を24区間から連結）
  video: duration = 26.830s   nb_frames = 1582  → 1582/60 = 26.367s ぶんの絵しかない
  audio: duration = 26.813s
  映像パケットPTSの全走査:
    20ms超のギャップ = 22箇所（＝区間境界の数）
    各ギャップ = 37.70ms （通常 16.67ms + 余分 21.03ms）
    余分の合計 = 22 × 21.03ms = 0.463s  ← 映像尺と絵の量の差に一致
```

- `subtitle_generator.py:572` のコメント「実10.0s のフレームがPTS≒10.4s」が指していたのはこの現象であり、**当時の推定（「区間ごとの微小な隙間が累積」）は正しかった**。その隙間の正体が **AAC 1フレーム（21.33ms）** であることが今回特定できた。
- **重要**：この時点の `silence_cut.mp4` は **A/V 同期している**。各フレームは正しい時刻に置かれており、境界で約21ms静止するだけである。**ズレは次工程で作られる。**

#### (4) `burn_subtitle` の `setpts=N/FRAME_RATE/TB` が階段を「直線」に潰す

`src/modules/subtitle_generator.py:603`：

```python
filter_spec = f"setpts=N/FRAME_RATE/TB,{scale_chain}{subtitles_opt}"
```

`setpts=N/FRAME_RATE/TB` は**入力の映像PTSを完全に破棄し、フレーム番号 N で等間隔に振り直す**。これは §2-4(3) の**不均一な階段状PTSを、一様な直線へ潰す**ことを意味する。

- 空白が密に入っていた箇所（＝カットが多い箇所）のフレームは、**本来より早い時刻へ前倒し**される。
- 音声は `-af aresample=async=1:first_pts=0` で実時刻を保持したまま出力される。
- テロップ（ASS）は**振り直した後の映像PTSの目盛り**へ描画されるため、**音声とは一致するが、映像の“絵”だけが先行**する。

これは §2-2 の観測（テロップ＝音声一致・映像のみズレ）と完全に整合する。

#### (5) 実測されたズレ量

26.8秒のスライスで、`setpts` 適用によって生じる映像のズレ（+ は映像先行）：

```
  true_pts =  0.021s → +21.0 ms   （冒頭から既にズレている）
  true_pts =  4.557s → +113.1 ms  （このスライスでの最大）
  true_pts = 26.684s → +23.7 ms
```

**全区間で正（映像先行）／冒頭から非ゼロ**であり、error.md の記述と一致する。

### 2-5. ズレの型：「尺に比例した累積」ではなく「カット密度に依存する変動ドリフト」

初版 §2-5 は「尺に比例して累積」としていたが、実測により**厳密には誤り**であることが判明した。

`setpts` は**全体平均の傾き**で直線化するため、ズレ量は「**局所の累積空白 − 全体平均の傾き × 経過時間**」となる。すなわち：

- ズレを決めるのは尺そのものではなく、**カット数（＝空白の総量）と、カット密度の時間的な偏り**である。
- カット密度が全体平均より高い区間で映像先行が拡大し、低い区間で縮む（符号が反転することもある）。

`ark.mp4` 全長に `silencedetect=noise=-23dB:d=0.6`（`setting.json` 実値）を適用して算出した結果（`verify.md` §4）：

```
無音区間 889件 / 有音区間(keep) 865件 / 残尺 1251.6s (20.9分)
映像PTS全長 1269.8s vs 実コンテンツ 1251.6s → 空白合計 18.2秒

最大ズレ = +0.928 秒（映像先行）
  出力  0.0分 →     +0 ms
  出力  2.5分 →   +300 ms
  出力 10.8分 →   +449 ms
  出力 14.5分 →   +827 ms   ← 最大付近
  出力 18.0分 →   +232 ms
  出力 21.1分 →    -23 ms
```

会話が多く細かく切られる箇所で映像先行が最大化する。**3時間・会話比率の高いアーカイブでは1秒超に達し得る**。

### 2-6. なぜ「別PC」で顕在化し、開発機で気付かなかったか

初版は「NVIDIA GPU＝VFR素材」で説明していたが、これは §2-3 で否定された。実測に基づく説明は以下。

- 本事象は **GPU にも素材の VFR/CFR にも依存せず、カット数が増えれば必ず発生する**。
- 開発機での検証素材は**短く、カット数が少ない**ためズレが数十ms程度に収まり、目視で気付けなかった。
- 実運用の3時間アーカイブは**カット数が桁違い**（`ark.mp4` の2時間で865区間）であり、かつ密度の偏りが大きいため、秒オーダーのズレとして表面化した。

すなわち **「カット数 × `setpts` 均等再配置」** が本質であり、PC・GPU の違いは**素材の長さとカット密度の違い**に還元される。

### 2-7. 副次的に判明した不具合（要対処）

1. **極短区間で音声ストリームが欠落する**
   本検証で、区間長 0.024秒 の区間から**音声ストリームを持たない区間ファイル**が生成された（AAC 1フレーム＝21.33ms 未満のため音声が1フレームも出力されない）。音声欠落ファイルを `-c copy` で連結すると、その位置で同期がさらに壊れる可能性がある。
   `build_keep_segments`（`silence_cutter.py:86`）の最小区間長は現行 **0.01秒** で、AAC 1フレーム長を下回っている。

2. **`silence_cut.mp4` の `r_frame_rate` が 240/1 に壊れる**
   境界の 37.70ms ギャップにより ffprobe のフレームレート推定が破綻し、`r_frame_rate=240/1` / `avg_frame_rate=58.964` となる（いずれも 60 ではない）。
   `setpts` の `FRAME_RATE` が `avg_frame_rate` と `r_frame_rate` のどちらを採るかは **ffmpeg のバージョン・ビルド依存**であり、**`r_frame_rate`(=240) が採られた場合は映像が4倍速になる**。「別PCで症状が異なる／悪化する」要因になり得るため、**暗黙値に依存しない書き方**（§3 対策A）が安全側である。

## 3. 修正方針

当初は「上流（無音カット）で空白を作らない＝根治」と「下流（焼き込み）で均等再配置をやめる＝安全化」の二層を想定したが、**上流側（対策B）を実測した結果、下流側（対策A）と同一の処理を前倒しで行うだけで追加コストしか生まないことが判明した**（`verify.md` §5・§6）。

したがって本書の方針は以下に確定する。

- **対策A（`setpts=N/FRAME_RATE/TB` → `fps={output_fps}`）を単独の主対策とする。**
- **対策B（上流での空白除去）は全案とも不採用**とし、判断根拠を記録として残す。
- **対策C（付随不具合の是正・診断ログ）を併せて実施する。**

いずれも `docs/CLAUDE.md` 方針（既存破壊なし・不要ライブラリなし・値は `setting.json` 管理）に従う。

### 対策A（主対策）：焼き込みの映像PTS再構築を「実時刻保持のCFR化」へ置換する

`burn_subtitle`（`subtitle_generator.py:603`）の `setpts=N/FRAME_RATE/TB` を、**実PTS基準で CFR 化する `fps` フィルタ**へ置き換える。

```python
# 案: フレーム番号ではなく実時刻に基づき CFR へ整える（境界空白の構造を保ったまま同期を維持）
filter_spec = f"fps={fps},{scale_chain}{subtitles_opt}"   # setpts=N/FRAME_RATE/TB を廃止
```

- `fps={fps}` は**各フレームの実PTSを見て**複製/間引きするため、**絵と実時刻の対応が保たれ**、音声（実時刻保持）とも整合する。実測で階段構造が保持されることを確認済み（`verify.md` §3）。
- `{fps}` は `setting.json` の `ffmpeg.output_fps` を明示的に渡すため、**§2-7(2) の `FRAME_RATE` 暗黙解決の環境差リスクも同時に解消**する。
- `-fps_mode cfr -r {fps}`・`-fflags +genpts`・`-af aresample=async=1:first_pts=0` は**維持**する。
- テロップ位置は「実PTS ＋ 実時刻由来の Whisper タイムスタンプ」で自然に一致するため、`setpts` による人為的な前ズレ補正は不要になる。
- 対策A では境界の空白が**複製フレームで埋められる**ため、カット箇所に約21msの静止が残る（`ark.mp4` 全長で計18.2秒ぶん＝1カットあたり1〜2フレーム）。**同期は正しくなり、視認もほぼ不可能な水準**である。
- **対策B の実測（下記）により、この処理を上流で行っても結果は同じで追加コストだけが増えることが判明した。したがって対策A を単独の主対策とする。**

### 対策B（上流・根治）：区間境界に空白を作らない — **実測の結果、いずれも不採用**

主因（§2-4(1)(2)）を上流で断つ案として以下を検討し実測した（`verify.md` §5・§6）。**結論として、対策A があれば対策B はいずれも不要**である。

1. **B-1: 区間抽出の音声を明示正規化する**（`extract_segment`）— ✗ 根治にならない
   `-ar {ffmpeg.audio_sample_rate}` と `-af aresample=async=1` を付与し、区間単位で映像・音声の実尺を揃える。
   → **AAC の1024サンプル量子化そのものは消えない**ため、緩和にとどまる。実測でも `-ar 48000` 付与後に音声−映像差が +5〜22ms とばらつくだけで、空白は残った。単独採用は不可。
2. **B-2: 連結を再エンコード化する**（`silence_cut.concat_reencode=true`）— ✗ **実測で無効と判明**
   `_concat_demux` の再エンコード分岐（`silence_cutter.py:359-360`）は `build_encode_options` を付けるだけで **`-fps_mode` も `-r` も指定していない**ため、入力の映像PTSがそのまま出力へ引き継がれる。
   実測（`verify.md` §5）: **ギャップ22件・空白 0.467秒が残存**（33.33ms/50ms へ丸められただけ）。`setpts` 適用時の最大ズレは **+114ms → +158ms と悪化**。処理時間は **+61%**（49.6s → 80.0s）。
   得られた効果は `r_frame_rate` の 240/1 → 60/1 正常化のみで、**同期は直らない**。
   - **B-2'（改良版）**: 上記の再エンコード分岐に **`-fps_mode cfr -r {output_fps}` を追加**すると、**ギャップ0件・空白 0.0000秒**・`r_frame_rate`=`avg_frame_rate`=60/1 となり完全に解消する（`setpts` を残しても最大ズレ +33ms＝約2フレーム）。
     ただし**これは対策A が焼き込み時に行う処理と同一**であり、残尺ぶんのフル再エンコードが1世代増える（`ark.mp4` 実測外挿で **約25分の追加時間と画質1世代分の劣化**）。**対策A を入れるなら重複投資となるため不採用**。
3. **B-3: `extract_mode="filter"` への切替**（`cut_and_concat` / `cut_and_concat_batched`）— ✗ 実行不能
   フィルタ内 `trim`＋`setpts=PTS-STARTPTS`＋`concat` で単一パス連続PTS化する案。原理的には本問題が起きないが、`trim` は**入力をフレーム0から全デコード**するため（`docs/error/20260611/resolve.md:11-15`）、実測したデコード速度 **約9.2倍速**（`ark.mp4` 1本フルデコード ≒ 13分）から次のジレンマに陥る。
   - `batch_size=50`（現行 `setting.json`）→ 865区間で **18バッチ ＝ フルデコード18回 ≒ 3.9時間**（実用不可）。しかも最後に `_concat_demux` を通るため**バッチ境界で同じ空白が再発**する（22箇所→17箇所に減るだけ）。
   - `batch_size<=0`（一括）→ フルデコードは1回で済み空白も原理的に発生しないが、**865分岐の split ファンアウト**となり、`docs/error/20260610/error2.md`（無音区間1159件・2時間素材で ffmpeg 無応答）の**ハング再発条件そのもの**。
   → 採用しない。なお `extract_mode` は `setting.json` / `DEFAULT_SETTINGS` / コード既定の3箇所すべてが `"seek"` で一致しており、GUI からは変更できない（`settings_window.py:1220` の保存対象外）。
4. **B-4: 区間音声をPCM化する** — △ 判定保留（優先度低）
   AAC の1024サンプル量子化を避けるため中間区間の音声を非圧縮にする案。PCM 格納のため `.mkv` を用いた実測では、**Matroska のコンテナ尺の丸めで空白がかえって 0.736秒へ拡大**した。mp4 で成立する形式を選ばない限り是非を判定できないため、対策A で解決する以上は追わない。

### 対策C（付随不具合の是正）

- **C-1**: `build_keep_segments` の最小区間長（現行 0.01秒）を、**AAC 1フレーム長（1024/`audio_sample_rate` ≒ 0.0213秒）以上**へ引き上げ、音声ストリーム欠落区間の生成を防ぐ（§2-7(1)）。閾値は `setting.json` 管理とするか固定計算とするかは §7-4。
- **C-2**: 焼き込み後の出力に対し、映像ストリーム尺・音声ストリーム尺・`avg_frame_rate`/`r_frame_rate` を INFO ログへ出す（`ffprobe` 一回）。両尺の差分がズレ量の一次指標になり、再発時の切り分けを即時化する。既存の `probe_duration`/`probe_dimensions` の枠内で追加し、過剰実装は避ける。

## 8. 検証方針

1. **再現（済）**：`ark.mp4` で映像PTSの境界空白（37.70ms × 境界数）と `setpts` によるズレを実測（`verify.md` §1〜§4）。
2. **対策B の棄却（済）**：`concat_reencode` の True/False を同一素材で比較し、**B-2 では空白が消えない**ことを実測（`verify.md` §5）。B-3 のコストも試算で棄却（`verify.md` §6）。
3. **対策A**：焼き込み出力で (a) 映像＝音声同期、(b) テロップ＝発話一致、(c) `drop`≒0・muxer エラー無し、を確認。生成コマンドは `progress.py` のデバッグログ（`-vf` が `fps=60,subtitles='...\:...'`）で確認する。
