# resolve.md 検証結果（実素材 `ark.mp4` による実測）

対象: `docs/error/20260719/resolve.md` §7（不明点・確認事項）／§8（検証方針）のうち、**動画のみで検証可能な項目**
使用素材: `D:/develop/autoedit/ark.mp4`（実際にズレが発生した入力）
検証日: 2026-07-20 / 検証環境: 本開発機（同梱 `src/ffmpeg/ffmpeg.exe`）

> **結論（先に要点）**
> - **resolve.md §2-3（VFRが主因）は成立しない。** `ark.mp4` は実測で **完全なCFR 60fps**（フレーム数＝実尺×60が誤差0.06フレームで一致）。
> - **真因は「AACフレーム量子化による区間音声の伸長（+約21.3ms/区間）と、concat デマルチプレクサの区間オフセット」**。これにより `silence_cut.mp4` の**映像PTSに区間境界ごと約21msの空白**が入る。
> - `burn_subtitle` の `setpts=N/FRAME_RATE/TB` は、この**不均一な階段状PTSを一様な直線へ潰す**ため、映像だけが音声・テロップに対して前後する。
> - `ark.mp4` の実カット構成で試算すると **最大 +0.93 秒の映像先行**、かつ**冒頭から非ゼロ**。error.md の症状（映像先行・最初からズレ・音とテロップは一致）と完全に一致する。
> - **上流での根治策（対策B）は全案とも不採用。** B-2 は空白が消えず（§5）、B-3 はフルデコード18回＝約3.9時間かハング再発（§6）。**対策A（`setpts` → `fps={output_fps}`）の単独適用が最も費用対効果が高い。**

## 1. §7-2「ソースが VFR か」→ **NO（CFR）**

```
ffprobe -select_streams v:0 ark.mp4
  codec=hevc(Main) 2560x1440  hev1
  r_frame_rate   = 60/1
  avg_frame_rate = 1646000000/27433341 = 59.99989
  nb_frames = 432898 / duration = 7214.9677s  → 59.99998 fps
  start_time(v) = 0.034000 / start_time(a) = 0.000000
```

- **フレーム数 432898 vs 実尺×60 = 432898.06** → 乖離 **0.06フレーム**。dup/drop 収支はゼロ。
- 10秒窓のパケット数を 0s / 1800s / 3600s / 5400s / 7100s で計測 → いずれも **600 フレーム**（先頭のみ 599＝start_time 0.034 の分）。局所的な VFR もなし。
- パケットPTSも 16/17ms 交互（＝1/60秒のms丸め）で等間隔。

→ **resolve.md §2-3 が前提とする「VFRソースの dup/drop でフレーム数と実尺が乖離する」現象は、この素材では起きていない。** 連動して **§2-4（NVIDIA GPU＝VFR素材という相関）も本件の説明にはならない**（NVENC収録だが記録はCFR）。

## 2. 実パイプライン再現（無音カット seek 方式）

`ark.mp4` の 3600s から 300秒を `-c copy` で切り出し（CFR 60fps・18061フレーム＝301.017s を確認）、
`src/modules/silence_cutter.py` の `detect_silence` → `build_keep_segments` → `cut_and_concat_seek` を**実コードのまま**実行（設定は `setting.json` 実値、`preset` のみ ultrafast＝タイムスタンプ挙動に影響なし）。

結果: 無音25件 → **有音24区間 / 残尺 26.17s**

### 2-1. 区間中間ファイル（`silence_seg_*.mp4`）の実測

| 項目 | 実測 |
|---|---|
| 映像尺 | 要求区間長 ≒ nb_frames/60（誤差 +0〜17ms、フレーム量子化のみ） |
| 音声尺 | **要求区間長 +21〜22ms が全区間で一定** |
| 音声−映像 | **常に +約21.3ms** |

**21.3ms = 1024サンプル / 48000Hz ＝ AAC 1フレーム**。区間ごとに音声が必ず1AACフレーム長く出力される。
（付随して: 最終区間 `silence_seg_00023.mp4`（区間長 0.024s）は **音声ストリームが存在しない**。極短区間で AAC フレームが1つも生成されないケースがあり、これも別途要注意。）

### 2-2. 連結後 `silence_cut.mp4` の実測 — **映像PTSに穴が空く**

```
video: duration=26.830s  nb_frames=1582  (1582/60 = 26.367s)
audio: duration=26.813s
r_frame_rate=240/1  avg_frame_rate=58.964   ← 60 ではない
```

映像パケットPTSの間隔を全走査した結果:

```
frames=1582  span=26.813s
gap>20ms の箇所: 22件（＝区間境界の数）
  各ギャップ = 37.70 ms  （通常 16.67ms + 余分 21.03ms）
  余分の合計 = 22 × 21.03ms = 0.463s
```

- 映像は **1582フレーム＝26.367秒ぶんの絵**しか持たないのに、**PTS上は 26.83秒に引き伸ばされている**（差 0.463s ＝ 空白の合計）。
- 原因: concat デマルチプレクサは**次ファイルのオフセットをファイル尺（＝長い方＝音声尺）で進める**ため、区間境界ごとに **映像側だけ約21msの空白**が生まれる。音声は連続。
- これは `subtitle_generator.py:572` のコメント「実10.0s のフレームがPTS≒10.4s」が指していた現象そのもの。**当時の推定（「区間ごとの微小な隙間が累積」）は正しく、その隙間の正体が AAC 1フレーム（21.33ms）である**ことが今回特定できた。

> 重要: **この時点の `silence_cut.mp4` は A/V 同期している。** 各フレームは正しいPTSに置かれており、境界で21ms静止するだけ。ズレは次工程で作られる。

## 3. §7-6「`setpts` → `fps` 置換の妥当性」→ **置換は妥当**

`silence_cut.mp4` に対し `burn_subtitle` と同一のオプション構成で両案を実行（字幕なし）:

| 案 | `-vf` | 出力 video | 出力 audio | frames |
|---|---|---|---|---|
| A（現行） | `setpts=N/FRAME_RATE/TB` | 26.833s | 26.819s | 1610 |
| B（提案） | `fps=60` | 26.850s | 26.819s | 1611 |

尺だけでは差が出ないが、**中身が違う**。`setpts` は `FRAME_RATE`（本環境では `avg_frame_rate`＝58.964）で**全フレームを等間隔に再配置**するため、§2-2 の**不均一な階段状PTSを一様な直線に潰す**。実測したズレ量:

```
silence_cut.mp4（26.8秒スライス）: setpts 適用で発生する映像ズレ
  true_pts= 0.021s  → +21.0 ms （冒頭から既にズレる）
  true_pts= 4.557s  → +113.1 ms（最大）
  true_pts=26.684s  → +23.7 ms
  ※ + は「映像が本来より早く表示される」＝映像先行
```

**全区間で正（＝映像先行）／冒頭から非ゼロ**。error.md の記述と一致する。
一方 `fps=60` は**各フレームの実PTSを見て**複製するため階段構造を保持し、絵と実時刻の対応が維持される。

## 4. §7-3「定数オフセットか累積ドリフトか」→ **どちらでもなく「カット密度に依存する変動ドリフト」**

`ark.mp4` 全長（2時間0分15秒）に対し `silencedetect=noise=-23dB:d=0.6`（`setting.json` 実値）を実行:

```
無音区間 889件 / 有音区間(keep) 865件 / 残尺 1251.6s (20.9分)
映像PTS全長 1269.8s vs 実コンテンツ 1251.6s → 空白合計 18.2秒
```

実測した境界あたり空白 21.03ms を用いて `setpts` 適用後のズレを算出:

```
最大ズレ = +0.928 秒（映像先行）
  出力  0.0分 →     +0 ms
  出力  2.5分 →   +300 ms
  出力 10.8分 →   +449 ms
  出力 14.5分 →   +827 ms  ← 最大付近
  出力 18.0分 →   +232 ms
  出力 21.1分 →    -23 ms
```

- ズレは**単調累積ではなく、カット密度が全体平均より高い区間で拡大し、低い区間で縮む**（`setpts` が「全体平均の傾き」で直線化するため）。
- 会話が多く細かく切られる序盤〜中盤で映像先行が最大化し、**最大 0.93秒**に達する。
- **尺そのものより「カット数」と「カット密度の偏り」が支配的**。3時間・会話比率の高いアーカイブなら 1秒超は容易に出る。

→ resolve.md §2-5「尺に比例して累積」は**厳密には誤り**。正しくは「**カット数に比例して空白が溜まり、その分布の偏りがズレ量を決める**」。

## 5. 対策B-2 の実測（`concat_reencode` の True/False 比較）

同一スライス（`ark.mp4` 3600s から300秒 / 24区間 / 残尺26.17s）に対し、`cut_and_concat_seek` を `concat_reencode` だけ変えて2回実行した。**`setting.json` の実値（`preset=medium` / `crf=20`）をそのまま使用**。

| 案 | 映像PTSの空白 | 20ms超ギャップ | `r_frame_rate` | `setpts`適用時の最大ズレ | 所要時間 |
|---|---|---|---|---|---|
| 現行（`concat_reencode=false`） | 0.463s | **22件** | 240/1（破綻） | **+114.0 ms** | 49.6s |
| **B-2**（`concat_reencode=true`） | 0.467s | **22件** | 60/1 | **+157.9 ms** | 80.0s |
| **B-2'**（B-2 + `-fps_mode cfr -r 60`） | **0.0000s** | **0件** | 60/1 | **+33.0 ms** | 78.6s |

### 5-1. B-2 が効かない理由

`_concat_demux` の再エンコード分岐（`silence_cutter.py:359-360`）は `build_encode_options` を付けるだけで、**`-fps_mode` も `-r` も指定していない**。そのため入力の映像PTSがそのまま出力へ引き継がれ、**22箇所のギャップが残る**（33.33ms / 50ms と 1/60秒の倍数へ丸められただけ）。

- 空白は 0.463s → **0.467s とほぼ不変**。`setpts` 適用時のズレは **+114ms → +158ms と悪化**。
- 処理時間は **+61%**（49.6s → 80.0s）。
- **得られた効果は `r_frame_rate` の 240/1 → 60/1 正常化のみ**（§2-7(2) の環境差リスクは消えるが、同期は直らない）。

→ **resolve.md 対策B-2 は不採用。**

### 5-2. B-2'（再エンコード分岐に `-fps_mode cfr -r {output_fps}` を追加）は完全に効く

- **ギャップ 0件・空白 0.0000秒**。実PTSを見て複製フレームで穴を埋めるため、映像が真の CFR 60fps になる。
- `r_frame_rate` = `avg_frame_rate` = 60/1 に正常化。この状態なら `setpts` を残しても実害はほぼ消える（最大ズレ +33ms ＝ 約2フレーム）。

**ただしこれは対策A（焼き込み時の `fps={output_fps}`）と同一の処理**であり、対策A を入れるなら重複投資になる。追加コストは残尺ぶんのフル再エンコード1世代（実測 1.2倍速 → `ark.mp4` の残尺20.9分に外挿して **約25分の追加時間と画質1世代分の劣化**）。

→ **対策A があれば不要。不採用。**

### 5-3. B-4（区間音声のPCM化）は判定保留

AAC の1024サンプル量子化を避けるため、中間区間の音声を `pcm_s16le`（`.mkv` 格納）にして同条件で実測した。

```
out_B4_pcm_copy.mkv : 空白 0.736s / ギャップ 22件 / 最大ズレ +145.8 ms
out_B4_pcm_enc.mp4  : 空白 0.733s / ギャップ 22件 / 最大ズレ +146.2 ms
```

**かえって悪化した。** 原因は PCM ではなく **Matroska のコンテナ尺の丸め**（区間長 0.702s に対し mkv のファイル尺が 0.750s ＝ +48ms。mp4+AAC の 0.738s より大きい）。PCM 案そのものの是非は mp4 で成立する形式を選ばないと判定できないため保留とする。対策A で解決する以上、追う必要はない。

### 6-2. デコード速度の実測とコスト

`ark.mp4` の映像デコード速度を実測 → **約9.2倍速**（2時間の1本フルデコード ≒ **13分**）。

`filter` 方式は `trim` を使うため入力をフレーム0から全デコードする（`docs/error/20260611/resolve.md:11-15`）。`ark.mp4` の実カット数 865区間で試算すると：

| `batch_size` | バッチ数 | フルデコード回数 | デコードのみの所要時間 | 問題 |
|---|---|---|---|---|
| 50（現行） | 18 | **18回** | **約3.9時間** | 実用不可。さらに最後に `_concat_demux` を通るため**バッチ境界で同じ空白が再発**（22箇所→17箇所に減るだけ） |
| 0（一括） | 1 | 1回 | 約13分 | **865分岐の split ファンアウト** → `docs/error/20260610/error2.md`（無音区間1159件・2時間素材で ffmpeg 無応答）の**ハング再発条件そのもの** |

→ **B-3 は「原理的に本問題が起きない」のが `batch_size<=0` の一括経路だけで、それがハングする構成**という板挟み。**不採用。**

## 7. resolve.md への修正提言（総括）

| resolve.md 記述 | 検証結果 | 提言 |
|---|---|---|
| §2-3 主因＝VFR × `setpts` 均等再配置 | **VFRではない（実測CFR）** | 主因を「**AAC 1フレーム(21.33ms)ぶん音声が長い区間ファイル × concat のファイル尺オフセット → 映像PTSの境界空白**」へ差し替え |
| §2-4 NVIDIA GPU＝VFR素材 | 成立しない（NVENC収録だがCFR） | 「別PCで発生」の説明は**素材のカット密度・カット数の違い**に置き換え（開発機の短い検証素材ではカット数が少なく気付けない） |
| §2-5 尺に比例して累積 | 部分的に誤り | 「**カット数に比例／密度の偏りで増減**」へ是正 |
| §2-6 `-ar` 未強制・AAC丸めは「寄与要因」 | **これが主因** | AAC丸めを寄与要因から**主因**へ格上げ |
| 対策A（`setpts` → `fps={fps}`） | **妥当。実測で階段構造を保持** | **単独の主対策とする。** 境界の21ms静止（1〜2フレーム）は残るが同期は正しく、対策B を足しても結果は同じ |
| 対策B-2（`concat_reencode=true`） | **✗ 無効（§5 で実測）** | 空白が残る。不採用 |
| 対策B-3（`extract_mode="filter"`） | **✗ 実行不能（§6 で試算）** | フルデコード18回 ≒3.9時間 or ハング再発。不採用 |

### 追加で判明した要確認事項

1. **極短区間で音声ストリームが欠落する**（本検証で区間長 0.024s の区間が音声なしで生成された）。音声欠落ファイルを concat すると同期がさらに壊れる可能性がある。`build_keep_segments` の最小区間長（現行 0.01s）を **AAC 1フレーム以上**へ引き上げる必要がないか要検討。
2. **`silence_cut.mp4` の `r_frame_rate` が 240/1 になる**（境界の 37.7ms ギャップが原因で推定が壊れる）。`setpts` の `FRAME_RATE` がどちらを採るかは ffmpeg のバージョン・ビルド依存であり、**もし `r_frame_rate`(=240) が採られれば映像が4倍速になる**。「別PC で症状が違う／悪化する」要因になり得るため、`FRAME_RATE` のような**暗黙値に依存しない**書き方（＝対策A の `fps={fps}` 明示）が安全。
